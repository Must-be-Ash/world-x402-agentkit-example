MPP: The Machine Payments Protocol

What is MPP?

MPP (Machine Payments Protocol) is an open payment protocol built on HTTP 402 Payment Required, designed to let any client — AI agents, apps, or humans — pay for services without API keys, OAuth flows, or billing accounts. It was created by wevm (the team behind wagmi and viem) and is on the IETF standards track at paymentauth.org.

The core idea: a server says “pay me first” via a standard HTTP response, and the client pays and retries. No signup, no accounts, no checkout pages.

MPP vs x402: How Do They Compare?

Both protocols use HTTP 402 Payment Required to gate resources behind payments. But they differ significantly in scope and design.

What’s Unique to Each

Only in x402

  • Facilitator model (offloads blockchain interaction to a shared service — simpler server-side setup)
  • Bazaar extension (service discovery and paid endpoint catalogs)
  • SIWX extension (wallet-based auth to exempt returning users from payment)
  • Go SDK with Gin middleware
  • Browser paywall component for human-facing UIs
  • Multi-chain EVM + Solana with broad L1/L2 coverage

Only in MPP

  • Payment-method agnostic (cards, Bitcoin Lightning, stablecoins, custom rails)
  • Session intent with real payment channels (off-chain vouchers, batch settlement)
  • SSE streaming payments with mid-stream voucher renewal
  • Permissionless extensibility (anyone can author new payment methods)
  • IETF standards track
  • Rust SDK
  • No facilitator dependency

Claude's take on if MPP is better than x402

It depends on what you’re building.

Choose MPP if:

  • You need to accept non-crypto payments (cards, wallets) alongside crypto
  • You’re building metered/streaming services (LLM APIs, token streaming) where session-based billing matters
  • You want to support multiple payment rails and let clients choose
  • You care about IETF standardization
  • You need Rust support

Choose x402 if:

  • You’re in the Coinbase/CDP ecosystem and want the simplest integration path
  • You only need USDC payments on EVM or Solana
  • You want the facilitator model to avoid handling blockchain interactions yourself
  • You need Go support
  • You want service discovery (Bazaar) or browser paywall components
  • You value the existing adoption and ecosystem

The big picture: MPP is more ambitious in scope — it’s trying to be the universal payment layer for HTTP, covering everything from credit cards to Bitcoin to stablecoins. x402 is more focused — it does crypto payments on EVM and Solana really well with a clean facilitator abstraction.

MPP’s session intent and streaming support are genuine differentiators if you’re building high-frequency metered services. x402’s facilitator model and broader chain support are genuine differentiators if you’re building in the crypto ecosystem.

They’re solving the same fundamental problem (HTTP 402 for machine payments) but making different tradeoffs. MPP optimizes for breadth of payment methods and streaming performance. x402 optimizes for simplicity of integration and breadth of blockchain support.

How Does MPP Work differently from x402?

Both MPP and x402 follow the same high-level concept: request → 402 → pay → retry → get resource. But the mechanics differ in important ways.

Headers: Standard vs Custom

x402MPP
ChallengePAYMENT-REQUIRED (custom header, JSON body)WWW-Authenticate: Payment (standard RFC 9110 auth-params)
CredentialPAYMENT-SIGNATURE (custom header)Authorization: Payment (standard HTTP auth header)
ReceiptPAYMENT-RESPONSE (custom header)Payment-Receipt (custom header)

MPP uses the standard HTTP authentication framework. x402 uses its own custom headers.

Verification: Facilitator vs Direct

This is the biggest architectural difference.

In x402, the server doesn’t verify payment itself. It forwards the signed payload to a facilitator (Coinbase hosts one at x402.org/facilitator), which does two separate steps:

  • POST /verify — validates the signature, checks balance, simulates the tx
  • POST /settle — executes the payment on-chain after the server does its work

In MPP, the server verifies payment directly using method-specific logic — no intermediary needed.

When Does Money Actually Move?

In x402: The client signs an authorization (EIP-3009 transferWithAuthorization), but money doesn’t move until the facilitator calls POST /settle after the server has done its work. It’s always a sign-then-settle model.

In MPP it depends on the method:

  • Tempo pull: Similar to x402 — client signs, server broadcasts the tx
  • Tempo push: Client sends the tx themselves, then presents the tx hash
  • Stripe: Payment happens when the server creates the PaymentIntent
  • Lightning: Client pays the invoice before retrying — the preimage revealed by the HTLC settlement is the credential

Challenge Format

x402 sends a JSON object with an accepts array — the client picks one PaymentRequirement from the list.

MPP sends auth-params in the WWW-Authenticate header and can include multiple headers for multiple payment options (e.g., one for Tempo, one for Stripe).

Status Codes

Both use 402 for the initial challenge. MPP additionally defines:

CodeMeaning
402Payment barrier — initial challenge, or credential was invalid/expired/malformed. Always includes a fresh challenge.
200Payment verified, resource granted. Includes receipt.
403Payment verified, but policy denies access (e.g., geo-restriction).
401Authentication failure unrelated to payment.

Payment Methods

MPP is payment-method agnostic. The core protocol doesn’t care how money moves — it only standardizes the challenge/credential/receipt exchange. Anyone can define a new payment method without needing approval from any foundation.

Built-in Methods

MethodWhat It IsNetworksIntents
TempoStablecoin payments on Tempo blockchainTempo mainnet/testnetCharge, Session
StripeCard payments via Shared Payment TokensTraditional card networksCharge
LightningBitcoin over Lightning NetworkLightning (mainnet, signet, regtest)Charge, Session
CardEncrypted network tokens (Visa, Mastercard)Card networksCharge

Tempo is the most fully-featured method — it supports both one-time charges and session-based streaming with ~500ms finality and sub-cent fees. It has two transaction modes: pull (client signs, server broadcasts — enables gas fee sponsorship) and push (client broadcasts directly).

Stripe uses Shared Payment Tokens (SPTs) — the client creates an SPT via Stripe’s API representing their card/wallet, sends it as the credential, and the server creates a PaymentIntent. Both sides need Stripe accounts.

Lightning uses BOLT11 invoices. For charges, the server generates an invoice, the client pays it, and the HTLC settlement reveals a preimage that serves as the credential. Verification is a single local sha256(preimage) == paymentHash check.

Card uses encrypted network payment tokens (e.g., from Visa Intelligent Commerce). The server provides an RSA public key in the challenge, and the client sends a JWE-encrypted token.

Custom Methods

Anyone can build a new payment method by defining a name, request/payload schemas (via Zod), a client createCredential function, and a server verify function. No governance approval needed.

How Does MPP Support Cards / Fiat?

MPP supports traditional card and fiat payments through two separate methods — Stripe and Card — each with a different architecture.

Stripe Method

Uses Shared Payment Tokens (SPTs) — a Stripe feature that lets a client share a payment method with a server’s Stripe account without exposing actual card details.

Flow:

  1. Server responds 402 with a challenge containing amount, currency, and Stripe details (Business Network profile, allowed payment method types)
  2. Client collects a payment method (via Stripe Elements or a stored card), then calls Stripe’s API to create an SPT with usage limits matching the challenge
  3. Client sends the SPT as the credential
  4. Server creates a Stripe PaymentIntent using the SPT and confirms it
  5. Server returns the resource + receipt referencing the PaymentIntent ID

Both client and server need Stripe accounts. SPT creation requires a Stripe secret key, so the client typically proxies this through a server endpoint.

Card Method

Uses encrypted single-use network payment tokens from card networks directly (e.g., Visa Intelligent Commerce). Unlike Stripe, the client and server can each use independent payment providers — no shared platform needed.

Flow:

  1. Server responds 402 with a challenge containing amount, accepted card networks, and an RSA public key (encryptionJwk)
  2. Client sends a card identifier to a credential issuer (e.g., Visa Intelligent Commerce)
  3. Credential issuer provisions a network token + cryptogram, encrypts them as a JWE (RSA-OAEP-256 + AES-256-GCM) using the server’s public key
  4. Client sends the encrypted token as the credential
  5. Server decrypts, charges through the card network via its own payment gateway
  6. Returns resource + receipt

Payment Intents: Charge vs Session

Charge Intent — One-Time Payment

One request, one payment, one receipt. The cost is known upfront.

Best for: per-call API endpoints, content access, tool invocations.

Session Intent — Pay-as-You-Go Streaming

This is where MPP gets interesting. Instead of paying per-request on-chain, the client opens a payment channel:

  1. Open: Client deposits funds into an on-chain escrow contract
  2. Use: For each request, the client signs an off-chain EIP-712 voucher saying “I’ve now consumed up to X total”
  3. Settle: The server periodically settles accumulated vouchers on-chain in batches
  4. Close: Remaining balance is refunded to the client

Verification is just an ecrecover call — microseconds, no RPC calls, no database lookups. The bottleneck becomes CPU, not blockchain TPS.

This gives you:

  • Sub-100ms per-request latency
  • Near-zero per-request fees
  • Thousands of requests on a single deposit

SSE Streaming with Mid-Stream Voucher Renewal

MPP natively supports per-token billing over Server-Sent Events. If the channel balance runs low mid-stream, the server emits a payment-need-voucher event and pauses. The client automatically signs a new voucher and sends it. The stream resumes on the same connection — no replay, no interruption.

This is purpose-built for LLM token streaming, metered APIs, and content aggregation.

MCP (Model Context Protocol) Transport

MPP maps natively onto MCP’s JSON-RPC layer for AI agent tool calls:

Credentialparams._meta["org.paymentauth/credential"] in tool call
Receiptresult._meta["org.paymentauth/receipt"] in response

This lets MCP servers monetize tool calls and AI agents pay autonomously.

Security Model

  • TLS 1.2+ required for all payment flows (1.3 preferred)
  • Challenge binding: Challenge IDs are HMAC-bound to all parameters — prevents tampering with payment terms
  • Single-use credentials: Each credential works exactly once — no replay attacks
  • Request body binding: For POST/PUT/PATCH, an optional digest parameter (RFC 9530) binds the challenge to the request body, preventing modification after challenge issuance
  • Idempotency: Idempotency-Key header supported for safe retries. Servers must not perform side effects on unpaid requests
  • Credential secrecy: Credentials are bearer tokens — must not be logged, cached, or included in error messages
  • Auth ordering: When both auth and payment are required, verify auth first (401), then issue payment challenge (402) — prevents leaking payment requirements to unauthenticated clients
  • Error handling: RFC 9457 Problem Details with seven specific error codes (payment-required, payment-insufficient, payment-expired, verification-failed, method-unsupported, malformed-credential, invalid-challenge)

Discovery: How Do You Find MPP Services?

MPP and x402 take different approaches to service discovery.

x402: Bazaar (Live Service Directory)

x402 has Bazaar — a working service discovery system. Resource servers advertise their paid endpoints in the 402 response via a bazaar extension, and facilitators index them into searchable catalogs. It’s a live directory of paid APIs you can browse and connect to today.

MPP: OpenAPI Discovery (Spec-Level, No Live Registry Yet)

MPP’s approach to discovery is defined in a formal specification but works differently from Bazaar. Instead of a centralized directory, each service publishes an OpenAPI document at /openapi.json that describes its payment capabilities.

What the OpenAPI document contains:

Each payable endpoint includes an x-payment-info extension with:

FieldPurpose
intent"charge" (per-request) or "session" (pay-as-you-go)
methodPayment method identifier (e.g., "tempo", "stripe")
amountCost in smallest denomination (e.g., cents), or null for dynamic pricing
currencyISO 4217 code (fiat) or token contract address (blockchain)
descriptionHuman-readable pricing note

Services can also include an x-service-info extension with categories and documentation links.

Important: Discovery is advisory only. The runtime 402 challenge is always authoritative — if the OpenAPI doc says one price and the 402 response says another, the 402 wins.

Registries and Aggregators (Spec’d but Not Built Yet)

The spec defines two additional layers that don’t exist yet in practice:

  • Registries: Crawl OpenAPI documents from submitted domains, validate the payment extensions, and index services into searchable catalogs. Re-crawl every 24+ hours, delist after 7+ consecutive failures. Domain ownership is proven by serving a valid OpenAPI doc with x-payment-info over HTTPS.
  • Aggregators: Consume registry data and add curation, trust scores, uptime metrics, or reshape it into agent-native formats. Can federate across multiple registries.

MPP Proxy Discovery Endpoints

The mppx proxy SDK does have built-in discovery for services routed through it:

EndpointDescription
GET /discoverLists all proxied services (JSON or markdown for AI agents)
GET /discover/{id}Details for a single service, including routes and pricing
GET /discover/allAll services with full route details
GET /llms.txtllms.txt-formatted overview of the proxy and its services

This is useful for agents interacting with a specific proxy, but it’s not a network-wide registry.

The Bottom Line on Discovery

MPP has discovery of payment methods and pricing per service (via OpenAPI), and specs out registries that would mirror Bazaar — but there’s no live registry implementation yet. x402’s Bazaar is ahead here with a working directory today. MPP’s mpp.dev site is documentation, not a service catalog.
